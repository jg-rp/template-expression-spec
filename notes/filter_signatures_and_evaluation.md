This approach makes the separation of concerns very clear: the engine handles the "plumbing" (evaluating expressions and passing values), while the filter handles the "logic" (interpreting those values).

### 3.4.3 Filter Signatures and Metadata

While filters are invoked dynamically, they may be accompanied by a **Signature**. This signature is a piece of metadata intended for tooling, static analysis, and ahead-of-time (AOT) validation.

- **Parse-Time Utility:** Signatures allow compilers or IDEs to provide autocomplete, verify argument counts, and check for the presence of required keyword arguments before the template is executed.
- **Decoupling:** The execution engine MUST NOT depend on this metadata to perform a filter invocation. The engine’s role is strictly to evaluate arguments and dispatch the resulting values to the filter implementation.
- **Polymorphism:** Because signatures are for tooling, a single filter name may be associated with multiple signatures (overloading) or a variadic signature without affecting the runtime evaluation model.

### 3.4.4 Argument Evaluation and Execution Model

The language employs an **eager evaluation** model for filter arguments.

#### Eager Evaluation

Before a filter is invoked, the engine evaluates the input expression and all provided argument expressions (positional and keyword) into concrete values.

- If an argument expression fails to evaluate, it resolves to `Nothing`.
- The filter is _always_ invoked, even if one or more arguments resolved to `Nothing` or `Null`.

#### Implementation Responsibility

The responsibility for validating and interpreting arguments lies entirely within the filter implementation. The engine does not perform automatic type coercion.

1. **Type Coercion:** If a filter requires a specific type (e.g., a `round` filter requiring a `Number`), the filter implementation must explicitly perform the coercion using the rules defined in **Section 2.4 (Type Conversion)**.
2. **Protocol Testing:** When a filter operates on complex types (like "Iterables"), it is responsible for testing the value at runtime. It must check if the value is a native `Array` or a `Drop` that implements the required protocol (e.g., the Sequence Protocol in 2.3.1).
3. **Missing Arguments:** If a filter requires a positional argument that was not provided in the template, the filter receives `Nothing` for that parameter slot.
4. **Totality:** In keeping with the requirement for total functions, if a filter receives arguments that are of an unsupported type or violate its internal logic, it MUST NOT panic. Instead, it SHOULD return `Nothing`.

---

### 3.4.7 Totality Requirement

To ensure template stability, all filters MUST be **total functions**. A filter is considered total if it returns a valid value for every possible combination of input and arguments.

- **No Exceptions:** A filter must never throw an exception or halt the execution of the template.
- **Fallback to Nothing:** If a transformation is mathematically or logically impossible (e.g., division by zero, or passing a `String` to a filter that only handles `Numbers` and cannot coerce the value), the filter MUST return `Nothing`.
- **Consistency:** The return value of a failed filter application should be indistinguishable from a variable lookup that yielded no result, allowing the `??` (Null Coalescing) operator to handle the fallback gracefully.
